Implement the core DFT (Dry Film Thickness) lookup engine -- the most critical and complex piece of the application.

<!-- CRITICAL: DFT Lookup Chain (verified against actual database) -->

The DFT lookup chain has been verified against the actual nullifire.db data.
The database does NOT have a simple "bands" table as the architecture doc assumed.
The ACTUAL chain is:

```
Step 1: Get Hp/A section factor
  - hp_profiles WHERE name = ? -> hp_over_a_band
  - section_factors WHERE steel_section_id = ? AND hp_over_a_band = ? -> hp_over_a, heated_perimeter

Step 2: Get coverage band mapping (use_band)
  - hp_data_info WHERE product_id = ? AND hour_id = ? AND composite_use_band = ? -> use_band
  - NOTE: composite_use_band is NOT always the same as hp_profile_name!
  - The 10 composite_use_band values are: C1, R1, R2, R3, S1, S2, S3, U1, U2, U4
  - Multiple hp_profiles map to the same composite_use_band
  - The mapping is: find the composite_use_band from hp_data_info that matches
    the hp_profile_name directly (if it's one of the 10), or find which 
    composite_use_band shares the same hp_over_a_band AND steel_type group
  - IMPORTANT: use_band can vary by fire_rating (hour_id)! E.g., SC601 U4: 
    hour_id=1 -> use_band='B', hour_id=3 -> use_band='C'

Step 3: Get coverage_id  
  - coverage_ranges WHERE band = use_band AND min_hpa = hp_over_a AND max_hpa = hp_over_a
  - This is an EXACT match (min=max=hp_over_a), not a range lookup
  - Hp/A values in section_factors are always multiples of 5
  - coverage_ranges has both wide ranges (min=5) and exact ranges (min=max)
  - Only exact ranges (min=max) have DFT data in dft_data table

Step 4: Get DFT
  - dft_data WHERE product_id = ? AND coverage_id = ? AND fire_rating_id = ? AND failure_temp_id = ?
  - Returns dft_mm (REAL)
  - Apply product.offset_dft if non-zero: final_dft = dft_mm + offset_dft

Verified example:
  254x254x73 UC (section_id=750) + U4 (UC 4-sided) + SC802 (product_id=266) + 60min (fire_rating_id=3) + 550C (failure_temp_id=7)
  -> hp_over_a_band='W', hp_over_a=155
  -> hp_data_info: composite_use_band='U4', use_band='C'
  -> coverage_ranges: band='C', min=max=155, coverage_id=720
  -> dft_data: dft_mm=0.367
```

hp_data_info schema: product_id, hour_id, product_type_id, composite_use_band, use_band, hp_use_default_band_id, maps_ft_band
  - hour_id maps to fire_rating_id in the fire_ratings table
  - composite_use_band values: C1, R1, R2, R3, S1, S2, S3, U1, U2, U4

Mapping hp_profile_name to composite_use_band:
  - If hp_profile_name IS one of {C1, R1, R2, R3, S1, S2, S3, U1, U2, U4} -> use directly
  - Otherwise: find which composite_use_band has the same hp_over_a_band as the hp_profile
    AND matches the steel type family. In practice:
    - U25 (UC 3-sided, band=U) -> U1 (same band=U, but note U1 is UB type 7 while U25 is UC type 8)
    - U22 (UC as beam 4-sided, band=W) -> U4 (UC 4-sided, same band=W AND same steel_type=8)
    - U20 (UB as column 4-sided, band=W) -> U2 (UB 4-sided, same band=W AND same steel_type=7)
    
  Reliable algorithm for mapping hp_profile to composite_use_band:
  1. If hp_profile_name is in the 10 direct values -> use it
  2. Get hp_profile's hp_over_a_band and steel_type_id
  3. Find composite_use_band whose hp_profile has same hp_over_a_band AND same steel_type_id
  4. If no match on steel_type_id (e.g., U25 type=8, but U1 type=7 shares band='U'):
     fall back to matching hp_over_a_band only

  Complete mapping (for intumescent, non-board-only profiles):
  hp_over_a_band -> composite_use_band(s):
    'U' -> U1 (3-sided UB/UC)
    'W' -> U2 (4-sided UB, type=7) or U4 (4-sided UC, type=8)
    'X' -> (partial -- not in hp_data_info, may not have DFT data)
    'I' -> C1 (circular)
    'AI' -> R1 or S1 (3-sided RHS/SHS)
    'AB' -> R2/R3 or S2/S3 (4-sided RHS/SHS)
    'Y','AE','AH' -> Angles (no composite_use_band entry -- may not be in hp_data_info)
    'T','AD','V' -> Channels (no composite_use_band entry -- may not be in hp_data_info)
    etc.

  For profiles WITHOUT a composite_use_band match, the DFT lookup returns status='no_data'.

Create `app/services/dft_service.py` implementing the verified DFT lookup chain.

**Constants:**
```python
# The 10 composite_use_band values that exist in hp_data_info
DIRECT_CUB = {'C1', 'R1', 'R2', 'R3', 'S1', 'S2', 'S3', 'U1', 'U2', 'U4'}
```

**`_resolve_composite_use_band(db, hp_profile_name: str) -> str | None`** (private helper):
1. If `hp_profile_name` is in `DIRECT_CUB`, return it directly
2. Get the hp_profile's `hp_over_a_band` and `steel_type_id`
3. Query all profiles whose `name` is in DIRECT_CUB: `SELECT name, hp_over_a_band, steel_type_id FROM hp_profiles WHERE name IN ('C1','R1',...)`
4. First try: find a DIRECT_CUB profile with same `hp_over_a_band` AND same `steel_type_id`
5. If no exact match: find a DIRECT_CUB profile with same `hp_over_a_band` (any steel type)
6. If still no match, return None

**`lookup_dft(section_id, hp_profile_name, product_id, fire_rating_id, failure_temp_id) -> dict`:**

Returns a dict with ALL of these fields (always present, some may be None):
```python
{
    "section_id": int,
    "hp_profile_name": str,
    "product_id": int,
    "fire_rating_id": int,
    "failure_temp_id": int,
    "hp_over_a": float | None,
    "heated_perimeter": float | None,
    "hp_over_a_band": str | None,
    "composite_use_band": str | None,
    "use_band": str | None,
    "coverage_id": int | None,
    "dft_mm": float | None,
    "offset_dft": float,
    "final_dft_mm": float | None,  # dft_mm + offset_dft
    "status": str,  # "ok", "no_section_factor", "no_band_mapping", "no_coverage", "no_dft_data", "error"
    "error": str | None
}
```

Implementation:
1. **Get hp_over_a_band from hp_profiles:**
   `SELECT hp_over_a_band FROM hp_profiles WHERE name = ?`
   If not found: return with status="error", error="Unknown HP profile"

2. **Get section factor:**
   `SELECT hp_over_a, heated_perimeter FROM section_factors WHERE steel_section_id = ? AND hp_over_a_band = ?`
   If not found: return with status="no_section_factor"

3. **Resolve composite_use_band:**
   Call `_resolve_composite_use_band(db, hp_profile_name)`
   If None: return with status="no_band_mapping"

4. **Get use_band from hp_data_info:**
   `SELECT use_band FROM hp_data_info WHERE product_id = ? AND hour_id = ? AND composite_use_band = ?`
   Note: `hour_id` in hp_data_info maps to `fire_rating_id` (they use the same IDs)
   If not found: return with status="no_band_mapping"

5. **Get coverage_id:**
   `SELECT coverage_id FROM coverage_ranges WHERE band = ? AND min_hpa = ? AND max_hpa = ?`
   Both min_hpa and max_hpa must equal hp_over_a (exact match)
   Cast hp_over_a to int before comparison (all values are whole numbers stored as REAL)
   If not found: return with status="no_coverage"

6. **Get DFT:**
   `SELECT dft_mm FROM dft_data WHERE product_id = ? AND coverage_id = ? AND fire_rating_id = ? AND failure_temp_id = ?`
   If not found: return with status="no_dft_data"

7. **Apply offset:**
   Get `offset_dft` from products table
   `final_dft_mm = dft_mm + offset_dft` (offset is 0.0 for all current products, but handle for future)
   Return with status="ok"

**Error handling:** Wrap entire function in try/except. On unexpected error, return status="error" with message. NEVER return a wrong DFT value -- it is better to return an error than an incorrect thickness.

**Important edge cases:**
- hp_over_a values are always multiples of 5 (e.g., 155, not 157)
- coverage_ranges exact match: `min_hpa = hp_over_a AND max_hpa = hp_over_a`
- Some product/fire_rating combos use different use_band than others (SC601 U4 at 30min uses 'B', at 60min uses 'C')
- If composite_use_band resolution fails, do NOT fall back to a guess -- return error

`POST /api/dft/lookup`:
- Accept JSON body: `{"section_id": int, "hp_profile_name": str, "product_id": int, "fire_rating_id": int, "failure_temp_id": int}`
- Validate all 5 fields are present, return 400 if any missing
- Call `dft_service.lookup_dft()` with the parameters
- Return the full result dict as JSON
- If status is "ok", HTTP 200
- If status is any error, still HTTP 200 (the error is in the data -- it's a valid lookup that found no result, not a server error)
- Only use HTTP 400 for malformed requests (missing fields, wrong types)

Why POST not GET: The lookup has 5 parameters that together form a compound key. A POST with JSON body is cleaner than a GET with 5 query params, and it matches the action semantics (computing a result, not fetching a resource).
